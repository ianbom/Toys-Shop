# Plan: Backend + detail notification per type

## Tujuan
Bangun data backend untuk `resources/js/pages/customer/notification/detail-notification.tsx` supaya halaman detail notifikasi tidak hardcoded, punya type berbeda minimal `payment` dan `shipping`, serta payload detail lengkap agar tampilan payment dan shipping bisa berbeda.

## Temuan
- `NotificationController::show()` sudah kirim props dari `NotificationService::detailData()`.
- `NotificationService::detailData()` masih kirim data sangat minim: hanya field notifikasi dasar + `reference_type/reference_id`.
- Notifikasi order-related selalu simpan `reference_type = 'order'` dan `reference_id = order.id` via `NotificationService::forOrder()`.
- Sumber data detail lengkap tersedia lewat relasi `Order -> payment`, `Order -> shipment`, `Shipment -> trackings`, `Order -> items`, `Order -> address`.
- Frontend detail page sekarang full statis dan hanya desain shipping.
- Frontend list page sudah kenal `payment`, `shipping`, `order`, dst. Jadi detail page tinggal menyesuaikan payload type.

## Implementasi

### 1. Perluas payload backend di `NotificationService`
Ubah `detailData()` dan mapper detail supaya kirim struktur lebih kaya:
- `notification` base:
  - `id`, `type`, `title`, `message`
  - `time`
  - `created_at`, `read_at`, `is_read`
  - `reference_type`, `reference_id`
  - `type_label` mis. `Payment Notification`, `Shipping Notification`
  - `status_badge` mis. `Read/Unread`
  - `is_important` boolean untuk type tertentu (`payment`, `shipping`)
- `detail_type`: normalized type untuk render page, mis. `payment`, `shipping`, fallback `general`
- `order`: ringkas tapi lengkap
  - `id`, `order_number`
  - `payment_status`, `order_status`, `shipping_status`
  - nilai uang: `subtotal`, `discount_amount`, `shipping_cost`, `service_fee`, `grand_total`
  - `created_at`, `paid_at`, `completed_at`, `expired_at`, `cancelled_at`
  - `notes`, `voucher_code`
  - `items_count`
  - `items` list ringkas: `id`, `product_name`, `product_image_url`, `quantity`, `subtotal`, `color_name`, `size`
  - `address` jika ada: recipient + full address
- `payment`: jika ada
  - `payment_provider`, `payment_method`, `transaction_status`, `fraud_status`, `gross_amount`, `currency`
  - `midtrans_order_id`, `midtrans_transaction_id`, `midtrans_redirect_url`
  - `paid_at`, `expired_at`
- `shipment`: jika ada
  - `shipping_provider`, `courier_company`, `courier_type`, `courier_service_name`
  - `delivery_type`, `waybill_id`, `biteship_order_id`, `biteship_tracking_id`, `label_url`
  - `shipping_cost`, `insurance_cost`, `estimated_delivery`, `shipping_status`
  - `shipped_at`, `delivered_at`, `cancelled_at`
- `timeline`: array event generik untuk right column timeline
  - gabung dari milestone order/payment/shipment sesuai type
  - shipping notification utamakan `shipment.trackings`
  - payment notification utamakan payment state changes + order milestones
- `actions`: action URLs/flags untuk frontend
  - `back_url = /notifications`
  - `order_url = /my-orders/{id}` bila route dipakai repo
  - `track_url` pakai shipment tracking/waybill jika memang ada halaman/URL, jika tidak `null`
  - `payment_url` pakai redirect Midtrans jika masih relevan, jika tidak `null`

### 2. Load relasi order sekali, aman, sesuai ownership
Di `NotificationService`, buat helper lookup reference order:
- Validasi notification milik user tetap dipertahankan.
- Jika `reference_type === 'order'` dan `reference_id` ada, query `Order` milik user dengan eager load:
  - `items`
  - `address`
  - `payment.logs`
  - `shipment.trackings`
- Sort tracking/timeline konsisten, idealnya descending untuk update terbaru di sidebar, atau ascending bila UI timeline dari lama ke baru. Pilih satu dan konsisten di transformer.

### 3. Buat mapper khusus per type
Di `NotificationService`, pecah logic supaya kecil dan rapi:
- `resolveDetailType(Notification $notification): string`
- `mapNotificationDetail(Notification $notification, ?Order $order): array`
- `mapOrderSummary(Order $order): array`
- `mapPaymentDetail(?Payment $payment): ?array`
- `mapShipmentDetail(?Shipment $shipment): ?array`
- `buildTimeline(Notification $notification, ?Order $order): array`

Aturan type:
- `payment` -> fokus data payment, nominal, method, status, invoice/ref IDs, payment timeline
- `shipping` -> fokus courier, waybill, ETA, shipping status, tracking timeline
- `order` -> fallback order summary + status timeline
- selain itu -> `general`, tetap bisa render pakai data notification base

### 4. Lengkapi controller bila perlu
Kemungkinan `NotificationController` tidak butuh perubahan besar.
Perubahan hanya bila ingin:
- tambah route model binding safety via scoped binding manual, atau
- biarkan service handle ownership seperti sekarang.

Paling minimal: tidak perlu ubah controller, cukup service.

### 5. Refactor `detail-notification.tsx` supaya data-driven
Ganti hardcoded UI dengan props typed dari backend.

Struktur frontend:
- Definisikan `Props` sesuai payload backend.
- Buat helper umum:
  - `formatPrice`
  - `formatDateTime`
  - `labelStatus`
  - config icon/color per notification type
- Header card dinamis:
  - icon, warna, label type beda untuk `payment` vs `shipping`
  - badge `Unread/Read`
  - badge `Important` untuk payment/shipping bila prop `is_important`
- Message Details card conditional:
  - untuk `shipping`: order no, shipping status, courier, service, ETA, tracking number, latest update
  - untuk `payment`: order no, payment status, provider, method, amount, paid at / expires at, transaction ID
  - fallback `general`: tampilkan metadata dasar
- Right column conditional:
  - shipping: related info courier/tracking/progress/ETA + tracking CTA
  - payment: related info provider/method/amount/status + payment CTA bila ada
- Timeline render dari `notification.timeline`
- Action buttons disesuaikan type:
  - back always
  - view order bila order ada
  - track shipment hanya untuk shipping dan jika shipment ada
  - complete payment / view payment hanya untuk payment dan jika URL ada
  - mark-as-read button boleh tetap tampil disable/hidden bila already read

### 6. Jaga kompatibilitas list notification
Tidak perlu ubah list page kecuali ingin samakan label/icon detail props. Fokus utama detail page.

## Validasi
Setelah implementasi:
- Buka detail notification type `shipping`:
  - page render tanpa data statis
  - tampil beda dari payment
  - courier, waybill, ETA, latest tracking muncul bila data ada
- Buka detail notification type `payment`:
  - page render beda layout/content
  - nominal, provider, method, status, transaction refs muncul
- Buka notification type lain (`order` / admin-created general):
  - page tetap aman render fallback
- Ownership check tetap 404 untuk notification user lain
- Mark as read tetap terjadi saat membuka detail
- Null-safe: order/payment/shipment/address/tracking kosong tidak bikin crash

## Risiko kecil
- URL order detail customer perlu cek route final. Bisa pakai string route existing bila sudah pasti, atau kirim `null` dulu bila ragu.
- Bila tidak ada sumber timeline historis payment selain current state + timestamps, timeline payment akan lebih ringkas dibanding shipping.
- Beberapa notifikasi admin manual mungkin punya `reference_type/reference_id` kosong; fallback `general` wajib aman.
